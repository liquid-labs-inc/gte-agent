## Known Limitations Carried Forward

1. Pre-existing core test failure: `Watcher > publishes .git/HEAD events` (macOS fs-events; fails on clean baseline). Still the only failing test in the tree (carried from the M6 record).
2. Codex backend fragility (plan's stated largest risk): `chatgpt.com/backend-api` is undocumented; the adapter's recorded fixtures are hand-authored pins of observed codex-CLI wire behavior and can drift if OpenAI changes the backend. Breakage degrades only the ChatGPT-OAuth path; the official API-key path is byte-for-byte unchanged.
3. Codex `session_id` is a random UUID per provider turn (GTE session ids are not UUIDs); if the backend prefers a stable per-conversation id, derive one from the session id later.
4. `ModelSelection` rewrites `~/.gte-agent/config.json` as plain JSON: it reads tolerantly (JSONC) but user comments in that file do not survive a selection write. Unknown keys are preserved.
5. Selecting an unauthenticated provider's model writes the global default before auth completes (deliberate — the TUI chains into the wizard), so abandoning the wizard leaves the default pointing at an unauthed model; the runner then surfaces the visible missing-credentials error directing to `/models`.
6. The TUI OAuth long-poll has no client-side abort: Esc invalidates the result via the epoch guard, but the underlying HTTP request lingers until the server-side wait times out (default 5 min). Harmless; the server discards stale flows on the next start.
7. The OAuth flow registry is in-memory and does not survive a daemon restart; a restarted sign-in must begin again at oauth/start (which also invalidates any previous pending flow for that provider).
8. Auth-route OpenAPI shows untyped request bodies for `auth.apiKey`/`auth.oauthComplete` — a deliberate trade-off: payloads decode from `Schema.Unknown` to defeat the value-echoing union the httpapi builder wraps around payload schemas (secret-safety over schema documentation). Other groups' payloads still use the default echoing decode path; any future secret-bearing endpoint must reuse the suppression pattern documented in `packages/server/src/groups/auth-provider.ts`.
9. The auth profile schema decode is strict: an unknown future profile type in an existing `auth.json` fails the whole-file decode (`InvalidAuthFileError`) rather than skipping the entry. Acceptable for the single-profile milestone; revisit if multi-profile lands.
10. Dead models.dev remnants left for cleanup (all no-ops after the catalog became static): unused env flags in `packages/core/src/flag/flag.ts` (`GTE_AGENT_MODELS_URL` / `GTE_AGENT_MODELS_PATH` / `GTE_AGENT_DISABLE_MODELS_FETCH`), the `GTE_AGENT_MODELS_DEV` define in `packages/cli/script/build.ts:79`, and the never-registered `packages/core/src/plugin/models-dev.ts`.
11. The generated JS SDK (`packages/sdk/js`) has not been regenerated for the new `/api/models` and `/api/auth` routes (the TUI uses thin raw-fetch clients matching the route contracts, per the existing `api/gte.ts` pattern). Run `./packages/sdk/js/script/build.ts` when the SDK needs them. Relatedly, optional response fields on these routes serialize as `null`, not omitted (effect httpapi behavior).
12. `Session.switchModel` in `packages/core/src/session.ts` still returns `OperationUnavailableError`; the models.select route calls `ModelSelection.select` directly.
13. The system prompt's "GTE environment" line degrades to "unknown" unless a `GteData` service is provided above the runner; current server composition provides it to the data handlers but not the session stack.
14. Carried unchanged from the M6 record: stub-auth cross-principal probe limitation, SDK branded types generating as `unknown`, upstream hyperliquid-dev `/health` 404, dash-form non-canonical symbol resolution, and stale pre-rename local dev DB state.
